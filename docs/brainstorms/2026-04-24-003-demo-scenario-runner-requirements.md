# Demo Scenario Runner — Seed Markdown + Playwright Filler

## Problem Frame

Every MCP demo script today (`scripts/mcp-demo-reasoning.mjs`, `scripts/mcp-demo-foaf.mjs`)
is a 300–400-line bespoke Playwright file that interleaves tool calls, result formatting,
and markdown generation in one blob. Adding a new demo means copying and rewriting the
whole script. When tools change, every script needs manual updates.

The goal: write the demo once as a readable **seed markdown** containing the conversation
narrative and tool call placeholders. A generic **runner script** fills the placeholders by
executing the tool calls against a live dev server, capturing real results and SVG snapshots.
The rendered output is identical in format to the existing relay-log demos.

---

## Concept: Seed + Runner

**Seed file** (`docs/mcp-demo/seeds/<demo-name>.md`):
- Written by hand once
- Contains the full user/assistant conversation narrative
- Tool calls are declared as fenced code blocks with a `tool-call` language tag
- Snapshot markers declare where SVG screenshots should be inserted
- This file is never overwritten by the runner — it is the source of truth

**Rendered file** (`docs/mcp-demo/<demo-name>.md`):
- Generated by the runner from the seed
- Placeholder blocks replaced with real tool results and SVG snapshots
- Never hand-edited — always regenerated

---

````markdown
# OWL-RL Reasoning Demo

> A recorded relay session. SVGs and tool results generated by `scripts/run-demo.mjs`.

---

**You:** I want to model a small organisation…

**Assistant:** Starting with four class nodes — I can batch them since they're independent.

addNode  {"iri":"http://example.com/reasoning-demo#Employee","typeIri":"http://www.w3.org/2002/07/owl#Class","label":"Employee"}
```

```tool-result

```

**You:** Now connect the subclass chain…

**Assistant:** Wiring the axioms:

```tool-call

```

```tool-result
<!-- runner fills this in -->
```

```snapshot

```

```tool-result
<!-- runner fills this in -->
```
````

## Requirements

**Seed format**

- R1. Tool calls are backtick-wrapped JSON-RPC 2.0 lines appearing inline in the assistant
  turn (not in a fenced block), matching the actual relay wire format:
  `` `{"jsonrpc":"2.0","id":<N>,"method":"tools/call","params":{"name":"<tool>","arguments":{...}}}` ``
  Consecutive such lines form a batch and execute sequentially.
- R2. Every `tool-call` block must be immediately followed by a `tool-result` block.
  The seed ships with `<!-- runner fills this in -->` as the placeholder. The runner
  replaces the entire content of that block; it never touches prose lines.
- R3. Snapshot markers are fenced blocks tagged `snapshot` containing exactly two
  YAML-style lines: `caption:` and `slug:`. The runner takes an SVG screenshot, writes it
  to `docs/mcp-demo/<demo-name>/<slug>.svg`, and replaces the snapshot block with a
  markdown image reference. The `snapshot` block itself is removed from the output.
- R4. Everything outside `tool-call`, `tool-result`, and `snapshot` blocks is copied
  verbatim into the rendered output — no transformation.

**Runner script** (`scripts/run-demo.mjs`)

- R5. Invocation: `node scripts/run-demo.mjs <seed-file> [--out <output-file>] [--url <base-url>] [--no-start-server]`.
  Default output is `docs/mcp-demo/<seed-basename-without-seeds-prefix>.md`.
  Default base URL is read from `vite.config.ts` (existing pattern).
- R6. The runner uses the existing Playwright + `window.__mcpTools` bootstrap already
  present in `mcp-demo-reasoning.mjs` (server start, page init, MCP polyfill injection,
  `registerMcpTools()` call). No new browser infrastructure — reuse and extract the shared
  bootstrap into a module.
- R7. For each `tool-call` block: parse all lines, call `window.__mcpTools[name](params)`
  sequentially, collect `{ toolName, params, result }` for each call.
- R8. The `tool-result` block content is replaced with a human-readable summary:

  ```
  [VisGraph — N tools ✓/✗]
  ✓ addNode: <iri>
  ✓ addLink: s=<subject> p=<predicate> o=<object>
  ✗ runLayout: <error message>
  
  Canvas: X nodes, Y links
  ```

  The canvas state line comes from a `getGraphState()` call after each batch.
- R9. For each `snapshot` block: call `fitCanvas` then `exportImage({format:'svg'})`,
  write the SVG to the output image directory, emit `![caption](./name/slug.svg)` in
  the rendered output.
- R10. The runner must NOT modify the seed file. Output always goes to the separate
  rendered path.
- R11. If a tool call returns `{ success: false }`, the runner logs the error to stdout,
  marks the result block with ✗, and continues (does not abort). The rendered file
  documents failures honestly.

**Shared bootstrap module** (`scripts/demo-bootstrap.mjs`)

- R12. Extract the Playwright setup logic shared across all demo scripts into a single
  module: server detection, server spawn, browser launch, page init, MCP polyfill
  injection, `registerMcpTools()` call. Returns `{ page, browser, cleanup }`.
- R13. Refactor the existing `mcp-demo-reasoning.mjs` and `mcp-demo-foaf.mjs` to use
  the shared bootstrap. The bespoke scripts keep their tool call sequences but drop the
  ~80 lines of repeated setup code.

## Success Criteria

- `node scripts/run-demo.mjs docs/mcp-demo/seeds/tbox-materials.md` produces
  `docs/mcp-demo/tbox-materials.md` with all `tool-result` blocks filled and SVG images
  written — no manual editing needed.
- Adding a new demo requires only writing a seed markdown file. No new script.
- The rendered output matches the existing relay-log format closely enough that a reader
  cannot tell whether the demo was hand-written or generated.
- `mcp-demo-reasoning.mjs` and `mcp-demo-foaf.mjs` still work after the bootstrap
  refactor — existing demos are not broken.

---

## Scope Boundaries

- The seed format is for relay-log style demos only — not for automated test assertions.
  Correctness verification (assertions) is a separate concern.
- The runner does not diff old vs. new output — it always regenerates from scratch.
- No vitest integration. The runner is a standalone Node/Playwright script.
- `tool-call` blocks execute sequentially within a block, and blocks execute sequentially
  in document order. No parallel execution in the runner.
- The runner does not interpret or validate IRI prefixes — params are passed verbatim to
  the MCP tools (the tools handle prefix expansion themselves).

---

## Key Decisions

- **Seed file is never overwritten:** The seed is the human-authored document; the rendered
  file is the machine-generated output. Keeping them separate prevents the runner from
  corrupting the narrative on re-run.
- **`tool-result` placeholder stays in the seed:** The `<!-- runner fills this in -->`
  comment means the seed is always valid markdown that renders sensibly on its own, even
  before the runner has been run.
- **Two spaces as tool-call separator:** `<name>  <JSON>` is unambiguous — tool names
  never contain spaces, JSON always starts with `{`. No special parser needed.
- **`snapshot` block is consumed (removed) not replaced:** The output image reference
  replaces it cleanly; a `snapshot` marker in the rendered file would be confusing.
