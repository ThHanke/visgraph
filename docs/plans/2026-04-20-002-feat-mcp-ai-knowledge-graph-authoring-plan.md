# feat: AI-native MCP knowledge graph authoring with chat demo

## Overview

Redesign the MCP tool layer so an AI agent can natively create, modify, and reason over knowledge graphs through natural-language chat. The current `navigator.modelContext` polyfill approach stays, but every tool is corrected or added to expose the full power of the app ŌĆö node/link authoring, expand-to-show-properties, viewport focus, SVG snapshot exports, OWL-RL reasoning, search, and Turtle export as the final deliverable.

A companion demo document (`docs/mcp-demo/foaf-social-network.md`) is generated by a Playwright test that runs a scripted AIŌĆōuser conversation, embeds SVG snapshots of each step, and produces linkable documentation for the README.

## Problem Frame

The app is a fully featured client-side knowledge graph editor. MCP tools exist but have several critical bugs and missing capabilities:

- `addNode` adds RDF triples but never calls `model.createElement` (now partially fixed) and never refreshes links or focuses viewport
- `removeNode` only deletes triples ŌĆö never removes the canvas element
- Newly added nodes are collapsed ŌĆö annotation properties invisible until double-clicked
- No `focusNode` / viewport-pan tool
- No `fitCanvas` tool
- `queryGraph` (SPARQL) is a stub
- Export of SVG for AI inspection is possible but not wired into the chat workflow
- The demo/documentation gap: no generated artifact showing the MCP workflow working end-to-end
- MCP tools bypass Reactodia's authoring state machine ŌĆö edits are written directly to rdfManager but are not staged through `editor.authoringState`, so they are not undoable and not persisted via `flushAuthoringState`. The correct pattern is to use `editor.setAuthoringMode(true)` + Reactodia authoring commands, then call `flushAuthoringState(editor, model)` to commit staged changes to the N3 store in one batch

## Requirements Trace

- R1. AI can load an ontology (FOAF, OWL, custom Turtle) to populate TBox/autocomplete
- R2. AI can create nodes with correct rdf:type and label, placed on canvas with links refreshed
- R3. AI can expand nodes to show annotation properties
- R4. AI can add directed links (triples) between existing canvas nodes
- R5. AI can remove nodes and links cleanly from both RDF store and canvas
- R6. AI can search/autocomplete to resolve IRIs before creating links
- R7. AI can run OWL-RL reasoning and refresh the canvas with inferred data
- R8. AI can focus the viewport on a specific node or fit the whole diagram
- R9. AI can export an SVG snapshot as a base64 string for user-facing display
- R10. AI can export the final graph as Turtle text
- R10b. Mutations via MCP are undoable (undo/redo) and persisted to N3 store via `flushAuthoringState`
- R11. A Playwright script generates `docs/mcp-demo/foaf-social-network.md` with embedded SVGs and a simulated chat transcript, linkable from README.md

## Key Technical Decisions

- **Use Reactodia authoring state for mutations**: `addNode`, `addLink`, `removeNode`, `removeLink` must go through `editor.authoringState` (via Reactodia authoring commands / `metadataProvider`) rather than calling `rdfManager.addTriple` directly. This gives undo/redo for free (`model.history`) and means `flushAuthoringState(editor, model)` correctly batches all staged changes into the N3 store. A new `saveChanges()` MCP tool calls `flushAuthoringState` explicitly ŌĆö or it is called automatically after each mutation. Decision: call `flushAuthoringState` at the end of every mutating tool so the store is always consistent.
- **SVG over PNG for AI tool results**: SVG is vector text ŌĆö far fewer tokens than a base64 PNG data URL. `exportSvg` returns a string directly. The AI can relay it as a fenced code block or inline image reference.
- **Auto-expand after addNode**: Every `addNode` call expands the element so annotation properties (labels, types, foaf:name etc.) are immediately visible. No separate user double-click needed.
- **Auto-refresh-links after addNode**: Call `model.requestElementData([iri])` + `model.requestLinks({ addedElements: [iri] })` after `createElement` so edges appear automatically.
- **Auto-focus after addNode and addLink**: After each mutation, call `focusNode(iri)` internally so the user sees what changed.
- **`fitCanvas` as standalone tool**: A dedicated zero-argument tool is simpler for the AI than passing empty args to layout.
- **New `expandNode(iri)` tool**: Explicit expand so the AI can expand any node on demand (e.g. after `loadRdf` when nodes were already on canvas).
- **`focusNode(iri)` new tool**: Pan + zoom to a specific IRI. Uses `zoomToFitRect` with padding.
- **Demo script is Playwright, not unit test**: It exercises the real live app, uses the polyfill, and captures real SVG exports. Output is a markdown file ŌĆö not a test assertion file.
- **`ctx.view.findAnyCanvas()` over `ctx.canvas`**: The research flagged `ctx.canvas` as unreliable early; use the safe accessor everywhere.

## System-Wide Impact

- **`navigator.modelContext` polyfill pattern**: tools only register when polyfill is injected or native API exists ŌĆö Playwright test must inject before app module load
- **`model.history.execute` for expand**: uses undo history, so expanded state is undoable by user ŌĆö acceptable side effect
- **`requestLinks` call signature**: must pass `{ addedElements: [...] }` not bare IRI array ŌĆö check Reactodia types
- **`ctx.view.findAnyCanvas()` vs `ctx.canvas`**: all tool files must migrate to the safe accessor
- **SVG export token cost**: a complex graph SVG can be 5ŌĆō15 KB of text ŌĆö still far cheaper than PNG base64. AI should call `fitCanvas` before export to ensure all nodes are in the viewport

## Risks & Dependencies

| Risk | Mitigation |
|------|------------|
| `Reactodia.setElementExpanded` import path unknown at runtime | Check Reactodia exports in `node_modules/@reactodia/workspace/dist/typings/` before implementing; fallback to direct property mutation if not exported |
| SVG export may include private/internal Reactodia CSS that breaks outside context | Use `addXmlHeader: true`; embed in `<details>` in markdown to isolate |
| Playwright polyfill injection timing (tools must register before app runs) | Use `addInitScript` Playwright API to inject polyfill before first script executes ŌĆö the existing demo approach uses post-load injection which is fragile |
| `model.requestLinks` call causes re-render loop if called too frequently | Debounce or batch calls within a single `addNode` call |
| Demo script depends on live network (foaf.rdf fetch fails in headless) | Use inline Turtle for FOAF data (as we discovered during testing); note this in script |

## Documentation / Operational Notes

- `docs/mcp-demo/foaf-social-network.md` is a **generated file** ŌĆö add to `.gitignore` or commit as a pre-generated artifact for the repo's demo purposes (prefer committed so it's browsable on GitHub)

- The polyfill approach (`navigator.modelContext` injection) is explicitly temporary ŌĆö when Chrome ships native AI APIs, this polyfill becomes unnecessary
